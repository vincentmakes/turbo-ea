# Глоссарий терминов

| Термин | Определение |
|--------|-------------|
| **ADR (Architecture Decision Record)** | Формальный документ, фиксирующий важное архитектурное решение, включая контекст, обоснование, последствия и рассмотренные альтернативы. ADR поддерживают процесс согласования и цепочку ревизий |
| **Статус утверждения** | Состояние проверки карточки: Черновик, Утверждено, Нарушено или Отклонено. Утверждённые карточки переходят в состояние «Нарушено» при редактировании |
| **Закладка / Сохранённое представление** | Сохранённая конфигурация фильтров, столбцов и сортировки в Инвентаризации, которую можно загрузить одним нажатием |
| **BPM** | Управление бизнес-процессами — дисциплина моделирования, анализа и улучшения бизнес-процессов |
| **BPMN** | Нотация моделирования бизнес-процессов — стандартная нотация для моделирования бизнес-процессов (версия 2.0) |
| **Бизнес-способность** | То, что организация может делать, независимо от способа реализации |
| **Вычисление** | Определённая администратором формула, автоматически вычисляющая значение поля при сохранении карточки |
| **Карточка** | Базовая единица информации в Turbo EA, представляющая любой архитектурный компонент |
| **Тип карточки** | Категория, к которой принадлежит карточка (например, Приложение, Бизнес-процесс, Организация) |
| **Показатель уверенности** | Оценка от 0 до 100%, показывающая, насколько надёжно сгенерированное ИИ описание |
| **Строка затрат** | Запись бюджета или фактических затрат (CapEx/OpEx) в инициативе PPM, используемая для отслеживания финансовых расходов |
| **Качество данных** | Оценка полноты от 0 до 100%, основанная на заполненных полях и их настроенных весах |
| **Диаграмма** | Визуальная архитектурная диаграмма, созданная с помощью встроенного редактора DrawIO |
| **Вложение файла** | Бинарный файл (PDF, DOCX, XLSX, изображения, до 10 МБ), загруженный непосредственно в карточку через вкладку «Ресурсы» |
| **DrawIO** | Встроенный инструмент для создания диаграмм с открытым исходным кодом, используемый для визуальных архитектурных диаграмм |
| **Корпоративная архитектура (EA)** | Дисциплина, которая организует и документирует бизнес- и технологическую структуру организации |
| **EOL (End of Life)** | Дата прекращения поддержки технологического продукта вендором. Отслеживается через интеграцию с endoflife.date |
| **Диаграмма Ганта** | Визуальная временная шкала с горизонтальными полосами, показывающая расписание, продолжительность и прогресс проекта |
| **Инициатива** | Проект или программа, включающая изменения в архитектуре |
| **Жизненный цикл** | Пять фаз, которые проходит компонент: Планирование, Внедрение, Активный, Вывод из эксплуатации, Окончание жизненного цикла |
| **LLM** | Большая языковая модель — модель ИИ, генерирующая текст (например, Ollama, OpenAI, Anthropic Claude, Google Gemini) |
| **MCP** | Model Context Protocol — открытый стандарт, позволяющий инструментам ИИ (Claude, Copilot, Cursor) подключаться к внешним источникам данных. Встроенный MCP-сервер Turbo EA обеспечивает доступ только для чтения к данным EA с RBAC на уровне пользователей |
| **Метамодель** | Управляемая данными модель, определяющая структуру платформы: типы карточек, поля, связи и роли |
| **Веха** | Значимое событие или точка завершения во временной шкале проекта, отображаемая как ромбовидный индикатор в диаграмме Ганта |
| **Уведомление** | Оповещение в приложении или по электронной почте, вызванное системными событиями (назначена задача, обновлена карточка, добавлен комментарий и т.д.) |
| **OData-канал** | JSON-канал данных, доступный для сохранённых представлений инвентаря (закладок) для использования внешними инструментами, такими как Power BI или Excel |
| **Ollama** | Инструмент с открытым исходным кодом для запуска LLM локально на собственном оборудовании |
| **Порядок строк BPM** | Порядок отображения строк типов процессов (Основные, Поддерживающие, Управленческие) в навигаторе процессов BPM, настраиваемый перетаскиванием строк |
| **Портфель** | Набор приложений или технологий, управляемых как группа |
| **PPM** | Управление портфелем проектов — дисциплина управления портфелем проектов и инициатив с бюджетами, рисками, задачами и отчётами о статусе |
| **Номер ссылки** | Автоматически генерируемый последовательный идентификатор для ADR (например, ADR-001, ADR-002), обеспечивающий уникальную, удобочитаемую метку |
| **Связь** | Соединение между двумя карточками, описывающее их взаимоотношение (например, «использует», «зависит от», «работает на») |
| **Вкладка «Ресурсы»** | Вкладка карточки, объединяющая записи архитектурных решений, вложения файлов и ссылки на документы в одном месте |
| **Статус RAG** | Индикатор здоровья Красный-Жёлтый-Зелёный, используемый в отчётах о статусе PPM для расписания, затрат и объёма |
| **Оценка риска** | Автоматически рассчитываемое значение (вероятность x воздействие), количественно определяющее серьёзность проектного риска |
| **Сохранённый отчёт** | Сохранённая конфигурация отчёта с фильтрами, осями и настройками визуализации, которую можно загрузить повторно |
| **Ревизия (ADR)** | Новая версия подписанного ADR, наследующая содержимое и ссылки на карточки из предыдущей версии, с увеличенным номером ревизии |
| **Секция** | Группируемая область на странице карточки, содержащая связанные поля, настраиваемая для каждого типа карточки |
| **Подписант** | Пользователь, назначенный для проверки и подписания документа ADR или SoAW. Процесс подписания отслеживает ожидающие и завершённые подписи |
| **SoAW** | Statement of Architecture Work — формальный документ TOGAF, определяющий объём и результаты инициативы |
| **SSO** | Single Sign-On — вход с использованием корпоративных учётных данных через провайдера идентификации (Microsoft, Google, Okta, OIDC) |
| **Подтип** | Вторичная классификация внутри типа карточки (например, Приложение имеет подтипы: Бизнес-приложение, Микросервис, ИИ-агент, Развёртывание). Каждый подтип выступает в роли подшаблона, который может управлять видимостью полей |
| **Шаблон подтипа** | Конфигурация видимости или скрытия полей для конкретного подтипа. Администраторы настраивают это в управлении метамоделью, нажимая на чип подтипа |
| **Заинтересованная сторона** | Лицо с определённой ролью в карточке (например, Владелец приложения, Технический владелец) |
| **Опрос** | Анкета для поддержания качества данных, нацеленная на определённые типы карточек для сбора информации от заинтересованных сторон |
| **Тег / Группа тегов** | Классификационная метка, организованная в группы с режимами одиночного или множественного выбора, необязательными ограничениями по типам и необязательным флагом обязательности, который блокирует утверждение и участвует в оценке качества данных |
| **Обязательная группа тегов** | Группа тегов, помеченная как обязательная. Применимые карточки нельзя утвердить, пока к ним не прикреплён хотя бы один тег из группы; её удовлетворение участвует в оценке качества данных карточки |
| **Семантическое обнаружение по Закону ЕС об ИИ** | Прогон TurboLens по соответствию, который просит LLM отметить карточки, встраивающие возможности ИИ / МО (LLM, рекомендательные системы, компьютерное зрение, скоринг, чат-боты …), даже если они не классифицированы явно как `AI Agent` / `AI Model`. Такие находки помечаются как **Обнаружено ИИ** |
| **Начальный vs остаточный риск** | Две оценки, фиксируемые для каждого риска в Реестре рисков. `Начальный` — вероятность × влияние без смягчения; `Остаточный` — вероятность × влияние после смягчения, редактируемые после появления плана смягчения. Оба выводят уровень через матрицу 4×4 |
| **Ссылка на риск** | Монотонный человекочитаемый идентификатор (`R-000123`), присваиваемый при создании риска. Остаётся видимым на кнопках продвинутых находок (**Открыть риск R-000123**) и в описании связанного Todo владельца |
| **Финансовый год** | 12-месячный период, используемый для бюджетирования и финансовой отчётности. Настраивается через Администрирование > Настройки — начальный месяц (с января по декабрь) определяет группировку бюджетных строк PPM |
| **TOGAF** | The Open Group Architecture Framework — широко используемая методология EA. Функция SoAW в Turbo EA соответствует TOGAF |
| **Отчёт о статусе** | Ежемесячный отчёт PPM, отслеживающий здоровье проекта через индикаторы RAG для расписания, затрат и объёма |
| **Веб-портал** | Публичное, доступное только для чтения представление выбранных карточек, доступное без аутентификации по уникальному URL |
| **Иерархическая структура работ (ИСР / WBS)** | Иерархическая декомпозиция объёма проекта на рабочие пакеты с датами начала/окончания и отслеживанием завершения |
| **Рабочий пакет** | Логическая группировка задач во временной шкале Ганта с собственными датами начала/окончания и процентом завершения |
| **ИИ-предложение** | Автоматически сгенерированное описание карточки, созданное путём объединения результатов веб-поиска с большой языковой моделью (LLM) |
| **ИИ-вердикт** | Подтверждение или отклонение пользователем классификации карточки LLM как содержащей ИИ (`hasAiFeatures = true / false`). Сохраняется при повторных сканированиях, чтобы дрейф LLM не менял неявно область действия AI Act ЕС |
| **GRC** | Governance, Risk и Compliance — единое рабочее пространство по адресу `/grc` с тремя вкладками (Governance, Risk, Compliance), объединяющее Принципы EA, ADR, Реестр рисков и сканер безопасности и соответствия |
| **Фаза G** | Фаза ADM «Implementation Governance» в TOGAF. Источник терминологии и жизненного цикла Реестра рисков |
| **Реестр рисков** | Ландшафтный реестр архитектурных рисков, выровненный с TOGAF Фазой G. Живёт по адресу `/grc?tab=risk`. Отличается от рисков уровня инициативы в PPM |
| **Владелец риска** | Пользователь, отвечающий за риск. Назначение автоматически создаёт системный Todo на странице задач владельца и отправляет уведомление `risk_assigned` |
| **Задача митигации** | Принадлежащий владельцу рабочий элемент, прикреплённый к Риску и фиксирующий конкретную митигационную активность. Может быть одноразовой или повторяющейся (ежедневно / еженедельно / ежемесячно / ежегодно). Повторяющиеся задачи корректно по календарю смещаются вперёд при закрытии |
| **Цикл задачи митигации** | Один запланированный экземпляр Задачи митигации. Проходит `scheduled` → `open` → `done` / `skipped`. Делает снимок назначенного при открытии и владельца при закрытии, чтобы аудиторские ответы переживали ротацию владельцев |
| **Время упреждения (Задача митигации)** | Дни до `due_date`, когда запланированный цикл повышается до `open` и попадает в список Todo назначенного. Умные значения по умолчанию для каждой единицы (1 / 2 / 7 / 14 для ежедневно / еженедельно / ежемесячно / ежегодно), ограниченные половиной длины цикла |
| **Находка соответствия** | Строка реестра Соответствия для регламента × статьи. Создана вручную рецензентом или произведена AI-сканированием TurboLens; оба типа разделяют один и тот же жизненный цикл и могут быть продвинуты в Риск |
| **Макро-возможность** | Группировка уровня 0 над L1 в Каталоге возможностей. Попадает в реестр как карточка `BusinessCapability` с `attributes.capabilityLevel = "Macro"` и `catalogueId` с префиксом `MC-`. Ослабляет ограничение глубины иерархии до 6 |
| **Слоистый вид зависимостей (LDV)** | Фирменная нотация Turbo EA для диаграмм зависимостей: карточки сгруппированы в четыре EA-слоя как плавательные дорожки, окрашены по типу карточки, предлагаемые карточки отображаются как узлы со штриховой границей и зелёным значком «NEW». Используется в отчёте о Зависимостях, разделе зависимостей детали карточки и целевой архитектуре TurboLens Architect |
| **TIME (Tolerate / Invest / Migrate / Eliminate)** | Четырёхступенчатая модель классификации портфеля приложений, популяризированная Gartner. Каждое приложение получает одну из позиций — Tolerate (оставить как есть), Invest (финансировать улучшения), Migrate (заменить или перенести) или Eliminate (вывести из эксплуатации). В Turbo EA обычно добавляется как поле `single_select` на типе Application и используется как цветовая ось в отчёте «Портфель» |
| **Рационализация портфеля приложений** | Самая распространённая стартовая EA-инициатива на Turbo EA: инвентаризация приложений в периметре, классификация по бизнес-ценности и технической пригодности и присвоение позиции TIME, определяющей решения по консолидации, замене или выводу из эксплуатации |
| **Crawl-Walk-Run** | Поэтапная модель внедрения, рекомендованная в Руководстве для начинающих EA. Crawl = узкий периметр, только Приложения, пять полей на карточку. Walk = добавить связь с возможностями и первый анализ портфеля. Run = расширение на процессы, интерфейсы, данные и продвинутые модули |
