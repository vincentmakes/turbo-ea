# Метамодель

**Метамодель** определяет всю структуру данных вашей платформы — какие типы карточек существуют, какие поля они содержат, как они связаны друг с другом и как организованы страницы карточек. Всё управляется **данными**: вы настраиваете метамодель через интерфейс администрирования, а не изменяя код.

![Настройка метамодели](../assets/img/ru/20_admin_metamodel.png)

Перейдите в **Администрирование > Метамодель** для доступа к редактору метамодели. Он содержит семь вкладок: **Типы карточек**, **Типы связей**, **Вычисления**, **Теги**, **Граф метамодели**, **Принципы EA** и **Регулирования соответствия**.

## Типы карточек

На вкладке «Типы карточек» перечислены все типы в системе. Turbo EA поставляется с 14 встроенными типами, распределёнными по четырём архитектурным уровням:

| Уровень | Типы |
|---------|------|
| **Стратегия и трансформация** | Цель, Платформа, Инициатива |
| **Бизнес-архитектура** | Организация, Бизнес-способность, Бизнес-контекст, Бизнес-процесс |
| **Приложения и данные** | Приложение, Интерфейс, Объект данных |
| **Техническая архитектура** | ИТ-компонент, Техническая категория, Поставщик, Система |

### Создание пользовательского типа

Нажмите **+ Новый тип**, чтобы создать пользовательский тип карточки. Настройте:

| Поле | Описание |
|------|----------|
| **Ключ** | Уникальный идентификатор (строчные буквы, без пробелов) — не может быть изменён после создания |
| **Название** | Отображаемое имя в интерфейсе |
| **Иконка** | Имя иконки Google Material Symbol |
| **Цвет** | Фирменный цвет типа (используется в инвентаре, отчётах и диаграммах) |
| **Категория** | Группировка по архитектурному уровню |
| **Поддержка иерархии** | Могут ли карточки этого типа иметь родительско-дочерние отношения |

### Редактирование типа

Нажмите на любой тип, чтобы открыть **панель деталей типа**. Здесь вы можете настроить:

#### Цвет типа

Каждый тип карточки — включая встроенные — имеет настраиваемый цвет, который используется в инвентаре, отчётах, представлениях зависимостей и диаграммах. Это позволяет привести Turbo EA в соответствие с визуальными конвенциями вашей организации (например, палитры TOGAF/ArchiMate: бизнес-элементы — жёлтый/оранжевый, приложения — синий).

- Выберите цвет с помощью образца цвета в панели. Если выбранный цвет имеет очень низкую контрастность на светлом или тёмном фоне, появится подсказка.
- У встроенных типов рядом с образцом цвета отображается кнопка **сброса**, когда цвет отличается от значения по умолчанию Turbo EA, — вы всегда можете вернуться к стандартной палитре.
- Текст поверх цветов типа (чипы, фигуры на диаграммах) автоматически переключается между чёрным и белым для читаемости, как в светлой, так и в тёмной теме.
- Рядом с палитрой отображается **живой предпросмотр**: имя типа, чип, значок карточки, подтип, бейдж идентификатора карточки и узел представления зависимостей — по одному разу в светлой и тёмной теме, с обновлением по мере выбора.

#### Поля

Поля определяют пользовательские атрибуты, доступные для карточек данного типа. Каждое поле имеет:

| Параметр | Описание |
|----------|----------|
| **Ключ** | Уникальный идентификатор поля |
| **Название** | Отображаемое имя |
| **Тип** | text, multiline_text, number, cost, boolean, date, url, single_select или multiple_select |
| **Варианты** | Для полей выбора: доступные варианты с названиями и опциональными цветами |
| **Обязательное** | Должно ли поле быть заполнено для оценки качества данных |
| **Качество данных** | Вклад каждого поля в оценку настраивается на панели **Качество данных** (см. ниже) |
| **Только для чтения** | Запрещает ручное редактирование (полезно для вычисляемых полей) |

Нажмите **+ Добавить поле**, чтобы создать новое поле, или нажмите на существующее поле, чтобы отредактировать его в **диалоге редактора полей**.

#### Секции

Поля организованы в **секции** на странице карточки. Вы можете:

- Создавать именованные секции для группировки связанных полей
- Устанавливать для секций **одноколоночный** или **двухколоночный** макет
- Организовывать поля в **группы** внутри секции (отображаются как сворачиваемые подзаголовки)
- Изменять порядок полей внутри секции перетаскиванием и перемещать поле в другую секцию с помощью действия **переместить**

Специальное имя секции `__description` добавляет поля в секцию «Описание» на странице карточки.

#### Идентификатор карточки

Включите **генерацию ID карточки**, чтобы присвоить карточкам этого типа стабильный, читаемый идентификатор (например, `APP-00001`). Он отображается в виде копируемой «таблетки» рядом с типом карточки, как необязательный столбец (с сортировкой и фильтрацией) в реестре, в экспортах Excel и в формулах вычисляемых полей (через `data.reference`).

**Номер всегда генерируется автоматически**; вы управляете только **префиксом**. При включении в виде текста показывается предлагаемый префикс (выведенный из имени типа, например `APP-`) — нажмите карандаш, чтобы изменить его. Номер настраивается двумя параметрами:

- **Начать с** — первый номер серии (по умолчанию `1`).
- **Мин. цифр** — ширина дополнения нулями (по умолчанию `5`), поэтому `1` отображается как `00001`. Это минимум; номера расширяются при превышении. **Пример** в реальном времени показывает первый идентификатор.

Идентификаторы **глобально уникальны, доступны только для чтения и никогда не используются повторно и не изменяются**. Последовательность номеров ведётся **по префиксу во всём рабочем пространстве**, поэтому два типа с одинаковым префиксом образуют единую непрерывную серию без коллизий. Как только у карточки этого типа появляется идентификатор, весь формат — префикс, начало и мин. цифр — блокируется (поля становятся только для чтения); генерацию по-прежнему можно отключить. Сохранение никогда не присваивает ID существующим карточкам; используйте отдельную кнопку **Сгенерировать ID**, чтобы заполнить их по требованию (с индикатором выполнения и подтверждением).

#### Оценка качества данных

Оценка **качества данных** карточки — это взвешенная мера её полноты. Каждый влияющий фактор — каждое поле и пять встроенных факторов — настраивается в одном месте: на вкладке **Качество данных** редактора типа карточки. (Редактор разбит на вкладки — Основное, Связи, Роли заинтересованных сторон и Качество данных, переводы доступны по значку в заголовке.)

Важность каждого фактора задаётся простым ползунком с четырьмя уровнями, который также показывает соответствующее число:

- **Игнорировать (0)** — полностью исключён из оценки.
- **Обычный (1)** — учитывается один раз (по умолчанию).
- **Важный (2)** — учитывается вдвое больше.
- **Критический (3)** — учитывается втрое больше.

На панели перечислены пять **встроенных факторов** — **Описание**, **Жизненный цикл** (задана ли хотя бы одна дата жизненного цикла), **Обязательные связи**, **Обязательные теги** и **Роли заинтересованных сторон** (каждая роль, определённая для типа, считается заполненной, когда назначена заинтересованная сторона) — а затем каждое поле, сгруппированное по своей секции, с тем же ползунком. Например, задайте **Жизненный цикл** как *Игнорировать* для типа, карточки которого закономерно никогда не содержат дат, чтобы они не получали штраф.

Полоса **состава оценки** вверху панели показывает долю каждого фактора в максимально возможной оценке, чтобы сразу видеть, какие факторы преобладают. В макете карточки на вкладке **Основное** каждое поле — а также встроенные секции «Описание», «Жизненный цикл» и «Связи» — показывает небольшой значок с текущим номером уровня, чтобы видеть вес, не покидая эту вкладку.

Изменение любой важности немедленно пересчитывает оценку всех существующих карточек этого типа. Новые поля по умолчанию имеют значение *Обычный*, поэтому учитываются в оценке сразу после добавления.

#### Подтипы (Под-шаблоны)

Подтипы выступают в качестве **под-шаблонов** внутри типа карточки. Каждый подтип может управлять тем, какие поля видны для карточек этого подтипа, при этом все поля остаются определёнными на уровне типа карточки.

Например, тип «Приложение» имеет подтипы: Бизнес-приложение, Микросервис, ИИ-агент и Развёртывание. Администратор может скрыть поля, связанные с серверами, для подтипа SaaS, поскольку они не актуальны.

**Настройка видимости полей по подтипам:**

1. Откройте тип карточки в администрировании метамодели.
2. Нажмите на любой чип подтипа, чтобы открыть диалог **Шаблон подтипа**.
3. Переключайте видимость полей с помощью переключателей — отключённые поля будут скрыты для карточек этого подтипа.
4. Скрытые поля исключаются из оценки качества данных, поэтому пользователи не штрафуются за поля, которые они не могут видеть.

Когда подтип не выбран для карточки (или тип не имеет подтипов), все поля видны. Скрытые поля сохраняют свои данные — если подтип карточки изменится, ранее скрытые значения будут сохранены.

#### Роли заинтересованных сторон

Определите пользовательские роли для данного типа (например, «Владелец приложения», «Технический владелец»). Каждая роль содержит **разрешения уровня карточки**, которые объединяются с ролью пользователя на уровне приложения при доступе к карточке. Подробнее см. в разделе [Пользователи и роли](users.md).

#### Переводы

Нажмите кнопку **Перевести** на панели инструментов drawer'а типа, чтобы открыть **Диалог переводов**. Здесь вы можете предоставить переводы для всех меток метамодели на каждом поддерживаемом языке:

- **Метка типа** — Отображаемое имя типа карточки
- **Подтипы** — Метки для каждого подтипа
- **Секции** — Заголовки секций на странице деталей карточки
- **Поля** — Метки полей и меток вариантов выбора
- **Роли заинтересованных лиц** — Названия ролей, отображаемые в интерфейсе назначения заинтересованных лиц

Переводы хранятся вместе с каждым типом карточки и разрешаются при рендеринге в соответствии с выбранным пользователем языком. Непереведённые метки возвращаются к значению на английском по умолчанию.

### Удаление типа

- **Встроенные типы** мягко удаляются (скрываются) и могут быть восстановлены
- **Пользовательские типы** удаляются безвозвратно

## Типы связей

Типы связей определяют допустимые соединения между типами карточек. Каждый тип связи указывает:

| Поле | Описание |
|------|----------|
| **Ключ** | Уникальный идентификатор |
| **Название** | Название в прямом направлении (например, «использует») |
| **Обратное название** | Название в обратном направлении (например, «используется») |
| **Тип источника** | Тип карточки на стороне «от» |
| **Тип цели** | Тип карточки на стороне «до» |
| **Кардинальность** | n:m (многие-ко-многим) или 1:n (один-ко-многим) |

Нажмите **+ Новый тип связи**, чтобы создать связь, или нажмите на существующую, чтобы отредактировать названия и атрибуты.

### Атрибуты связи

Некоторые связи содержат дополнительные атрибуты, которые задаются для каждого отдельного соединения, а не для типа связи. Например, встроенная связь **Организация → Приложение** («использует») имеет атрибут **Тип использования** — задайте для каждого соединения значение **Владелец**, **Пользователь** или **Заинтересованная сторона**. Так вы сможете смоделировать приложение, которое *принадлежит* одной организации и *используется* другими, с помощью единственного типа связи. Выбранное значение отображается цветной меткой в разделе **Связи** карточки; задайте его при добавлении связи или позже с помощью значка редактирования в строке связи.

Между заданной парой типов карточек может существовать только один тип связи, поэтому используйте эти атрибуты для уточнения смысла соединения, а не создавайте второй тип связи для той же пары источника и цели.

### Управление значениями связи

Нажмите значок **Управление значениями связи** (ярлык) в строке связи, чтобы изменить значения её атрибутов «типа». Вы можете:

- **Добавить собственные значения** в существующий список — например, новый Тип использования помимо Владельца / Пользователя / Заинтересованной стороны.
- **Добавить совершенно новый список типов** к связи, у которой его нет, с помощью кнопки **Добавить тип** — в том числе для встроенных связей.

Встроенные значения (Владелец, Пользователь, Заинтересованная сторона, значения направления потока…) **заблокированы**: их нельзя переименовать, перекрасить или удалить. Однако встроенное значение можно **скрыть**, чтобы оно больше не отображалось в списке на карточках, — уже установленное значение остаётся видимым. Ваши собственные значения полностью редактируются и удаляются.

## Принципы EA

Вкладка **Принципы EA** позволяет определить архитектурные принципы, которые регулируют ИТ-ландшафт вашей организации. Эти принципы служат стратегическими ориентирами — например, «Повторное использование прежде покупки прежде разработки» или «Если покупаем — покупаем SaaS».

Каждый принцип имеет четыре поля:

| Поле | Описание |
|------|----------|
| **Название** | Краткое имя принципа |
| **Формулировка** | Что гласит принцип |
| **Обоснование** | Почему этот принцип важен |
| **Последствия** | Практические следствия соблюдения принципа |

Принципы можно **активировать** или **деактивировать** индивидуально с помощью переключателя на каждой карточке.

### Импорт из Каталога принципов

Turbo EA поставляется с **подобранным справочным каталогом из 10 отраслевых эталонных принципов EA**, чтобы вам не пришлось начинать с чистого листа. Откройте меню аватара в правом верхнем углу и выберите **Справочные каталоги → Каталог принципов**. Оттуда вы можете:

- Искать и просматривать включённые принципы (заголовок, описание, обоснование, последствия).
- Выбирать несколько записей и нажимать **Импортировать** — выбранные принципы появятся на вкладке «Принципы EA» как обычные, полностью редактируемые записи.
- Безопасно повторно импортировать: уже существующие принципы (распознаваемые по их стабильному ID каталога) пропускаются, даже если вы переименовали их локально. Такие записи в каталоге помечаются зелёным значком «Уже импортирован».

Используйте каталог как отправную точку, а затем адаптируйте заголовок, формулировку, обоснование и последствия каждого принципа под вашу организацию.

### Как принципы влияют на ИИ-аналитику

При генерации **ИИ-аналитики портфеля** в [отчёте по портфелю](../guide/reports.md#ai-portfolio-insights) все активные принципы включаются в анализ. ИИ оценивает данные вашего портфеля в соответствии с каждым принципом и сообщает:

- Соответствует ли портфель принципу или **нарушает** его
- Конкретные данные в качестве доказательств
- Рекомендуемые корректирующие действия

Например, принцип «Покупать SaaS» побудит ИИ отметить приложения, размещённые локально или на IaaS, и предложить приоритеты миграции в облако.

## Граф метамодели

![Граф метамодели](../assets/img/ru/38_graf_metamodeli.png)

Вкладка **Граф метамодели** отображает визуальную SVG-диаграмму всех типов карточек и их типов связей. Это визуализация только для чтения, которая помогает понять взаимосвязи в вашей метамодели на одном экране.

## Регулирования соответствия

Вкладка **Регулирования соответствия** управляет нормативными фреймворками, по которым работает [сканер Соответствия в GRC](../guide/grc.md#compliance). По умолчанию включены шесть фреймворков:

| Регулирование | Область |
|---------------|---------|
| **EU AI Act** | Требования к системам ИИ / МО, размещаемым на рынке ЕС |
| **GDPR** | Общий регламент ЕС по защите данных |
| **NIS2** | Директива ЕС о сетевой и информационной безопасности 2 |
| **DORA** | Регламент ЕС о цифровой операционной устойчивости для финансовых организаций |
| **SOC 2** | Критерии AICPA Service Organization Controls Trust Services |
| **ISO/IEC 27001** | Стандарт систем управления информационной безопасностью |

В каждой строке вы можете:

- **Включить / выключить** регулирование переключателем — отключённые фреймворки пропускаются при каждом следующем сканировании, а их находки исключаются из дашбордов. Существующие находки сохраняются (не удаляются) — на случай, если позже вы их снова включите.
- **Редактировать** заголовок, описание области и контекст подсказки, передаваемый LLM.
- **Добавить пользовательское регулирование** через **+ Новое регулирование** — например, HIPAA, внутренние политики или отраслевые фреймворки. Пользовательские регулирования полноправны: они появляются во вкладке по регулированию, вносят вклад в общий балл соответствия и поддерживают те же действия над находками (подтвердить, принять, продвинуть в Риск).
- **Удалить** пользовательское регулирование — встроенные регулирования удалить нельзя, можно только выключить.

Сканер соответствия и процесс продвижения в Риск работают **даже без настроенного ИИ-провайдера** — ручной ввод находок, переходы статусов и путь продвижения в Риск остаются доступными. ИИ требуется только тогда, когда вы реально запускаете новое сканирование.

## Редактор макета карточки

Для каждого типа карточки раздел **Макет** в панели деталей типа управляет структурой страницы карточки:

- **Порядок секций** — Перетаскивайте секции (Описание, EOL, Жизненный цикл, Иерархия, Связи и пользовательские секции) для изменения их порядка
- **Видимость** — Скрывайте секции, не относящиеся к данному типу
- **Раскрытие по умолчанию** — Выбирайте, будет ли каждая секция изначально развёрнута или свёрнута
- **Макет колонок** — Установите 1 или 2 колонки для каждой пользовательской секции
- **Перемещение полей между секциями** — Используйте действие **переместить** у поля (рядом с кнопками редактирования и удаления), чтобы перенести его в другую секцию с сохранением настроек

## Ресурсы

Вкладка **Ресурсы** управляет двумя списками, предлагаемыми на вкладке **Ресурсы** каждой карточки:

- **Типы ссылок** — категория ссылки на документ (например, *Документация*, *Договор*, *Безопасность*). Каждый тип ссылки также имеет **значок**, отображаемый рядом со ссылкой.
- **Категории файлов** — категория, назначенная загруженному вложению.

Для каждого списка вы можете:

- **Добавить элемент** — с ключом (идентификатор в нижнем регистре, сохраняемый в карточках и неизменяемый после создания), отображаемым названием и — для типов ссылок — значком.
- **Изменить** название, значок, порядок сортировки и переводы по языкам любого элемента, включая встроенные.
- **Включить / отключить** элемент переключателем — отключённые элементы исчезают из списка выбора, но существующие значения на карточках сохраняются.
- **Удалить** пользовательский элемент — встроенные элементы нельзя удалить, только отключить.

Встроенный тип ссылки **Договор** включён по умолчанию. Оба списка включены в **Перенос рабочей области**, поэтому ваши настройки клонируются между экземплярами.
