# Кастомизируйте метамодель — слегка

Метамодель Turbo EA полностью **настраиваема администратором** — каждый тип карточки, поле, подтип, связь и роль стейкхолдера — это данные, а не код. У Вас возникнет соблазн её переработать. **Не делайте этого.**

Команды, которые преуспевают, кастомизируют метамодель **только когда стандартные поля не могут ответить на их вопрос**. Команды, которые терпят неудачу, тратят первый месяц на переименование `Application` в `Solution`, добавление 30 пользовательских полей и так и не доходят до рабочего отчёта.

## Тест из двух вопросов перед добавлением поля

Прежде чем добавить хоть одно пользовательское поле, спросите себя:

1. **Буду ли я фильтровать, группировать или строить отчёт по этому полю?** Если нет, оно принадлежит описанию или тегу — а не полю.
2. **Нужен ли один и тот же ответ на каждой карточке этого типа?** Если нет, это связь или вложение, а не поле.

Если Вы не можете ответить «да» на оба, не добавляйте поле.

## Разобранный пример: добавьте TIME-диспозицию

Для Рационализации портфеля приложений Вам нужно одно решение на приложение: **T**olerate (Терпеть) / **I**nvest (Инвестировать) / **M**igrate (Мигрировать) / **E**liminate (Ликвидировать) (фреймворк **TIME**, популяризированный Gartner). Встроенная метамодель не поставляет поле `timeDisposition`, поэтому это один из редких случаев, когда добавление пользовательского поля — правильный шаг.

Мы добавим его как поле `single_select` на тип `Application`, с четырьмя цветными опциями, весом 1, чтобы оно влияло на качество данных.

### Шаг 1 — откройте редактор типа

1. Перейдите в **Админ → Метамодель**.
2. Кликните по карточке типа **Application**.
3. Справа откроется панель типа. Переключитесь на вкладку **Fields**.

### Шаг 2 — добавьте поле

1. Выберите секцию, в которую Вы хотите поместить поле (или создайте новую секцию под названием «Portfolio Decision»).
2. Нажмите **+ Add field** в этой секции.
3. Заполните:
    - **Key**: `timeDisposition`  *(lower camel-case, без пробелов, становится ключом атрибута в JSON)*
    - **Label**: *Portfolio Disposition (TIME)*
    - **Type**: `single_select`
    - **Weight**: `1`  *(влияет на показатель качества данных)*
    - **Required**: оставьте **выключенным** — required заблокирует согласование каждой существующей карточки.
4. Добавьте четыре опции:

    | Key | Label | Цвет |
    |-----|-------|--------|
    | `tolerate` | Tolerate | серый / нейтральный |
    | `invest` | Invest | зелёный |
    | `migrate` | Migrate | янтарный |
    | `eliminate` | Eliminate | красный |

5. **Добавьте переводы** для метки и каждой опции в каждой локали, которую Вы поддерживаете — страница 4 [Админ → Метамодель](../admin/metamodel.md) описывает редактор переводов. Пропуск этого означает, что неанглоязычные пользователи увидят «timeDisposition» дословно.
6. Сохраните.

### Шаг 3 — убедитесь, что работает

1. Откройте любую карточку Приложения. Новое поле появляется в своей секции, пустое.
2. Выберите значение, сохраните. Кольцо Качества данных должно подняться на несколько процентов.
3. Вернувшись в **Инвентаризация**, поле теперь доступно во вкладке **Columns** и как фильтр — Вы уже можете фильтровать приложения по TIME.

И всё. Одно поле, десять минут, сразу полезно.

## Альтернатива: используйте Tag Group вместо этого

Если значение информационное, а не запрашиваемое, **Tag Group** (Админ → Tags) легче, чем пользовательское поле — без изменения метамодели, без миграции, проще эволюционировать. Используйте Tag Group, когда:

- Значение описательное («Customer-facing», «Internal-only», «Acquired in 2024»).
- Вы можете часто добавлять новые опции.
- Вам не нужно это в фильтре-выпадашке, но чип тега с поиском по мере ввода подойдёт.

Используйте пользовательское поле, когда:

- Вам нужно значение на осях Portfolio Report (X, Y, цвет).
- Вы хотите взвесить его в Качество данных.
- Это контролируемый словарь, который не будет часто меняться.

TIME-диспозиция относится к лагерю пользовательских полей, потому что мы будем использовать её как цветовую ось Portfolio Report на следующей странице.

## Антипаттерны, которых следует избегать

Это самые распространённые ошибки в метамодели при первых развёртываниях:

!!! warning "Не переименовывайте встроенные типы карточек"
    Переименование `Application` в `Solution` выглядит аккуратно, но ломает концептуальное сопоставление, которое предполагают Capability Heatmap, Portfolio Report и каталоги. Если Ваша организация называет их «Solutions», установите перевод **label** — базовый `key` остаётся `Application`.

!!! warning "Не добавляйте 30 пользовательских полей в первый день"
    Каждое пользовательское поле добавляет трения в сбор данных и разбавляет показатель Качества данных. Добавьте одно поле, используйте его месяц, затем добавьте следующее.

!!! warning "Не делайте новые поля `required` в первый день"
    `Required` блокирует согласование для каждой существующей карточки, у которой нет значения. Делайте поле обязательным только **после** того, как заполните его для 80%+ популяции.

!!! warning "Не создавайте пользовательские типы карточек вместо пользовательских полей"
    «Mobile App» должен быть подтипом `Application`, а не новым типом карточки. Новые типы не получают сопоставление с возможностями, отчёты по портфелю или импорт из каталога бесплатно.

## Другие лёгкие расширения, которые Вам могут понадобиться

Это распространённые расширения второго прохода, но **не добавляйте их, пока они действительно не нужны**:

| Потребность | Куда добавить | Тип |
|------|-------------|------|
| Рейтинг бизнес-ценности | Application | `single_select` (High/Medium/Low) — управляет осью Y Portfolio Report |
| Рейтинг технической пригодности | Application | `single_select` — управляет осью X |
| Готовность к облаку | Application | `single_select` (Ready / Needs refactor / Stays on-prem) |
| Категория риска потери | Application, IT Component | `single_select` (Single point of failure и т. д.) |
| Разделение затрат | Application | поля `cost` для `costRunTotalAnnual`, `costChangeTotalAnnual` |

Каждое из них проходит тест из двух вопросов для аналитики портфеля. Каждое из них также является хорошим кандидатом на вычисляемую формулу вместо ручного ввода — что и рассмотрено на следующей странице.
